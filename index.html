Welcome to Cerberus.

Cerberus is an application providing an authentication service for people working in your company. It has been designed following the OAuth2 protocol which allows a client application to authenticate an end-user by simplifying the authentication flow. In most cases the user does not have to enter his credentials anymore!

What is behind the scenes?

This application is built using the famous Ruby On Rails framework. The process used is two-fold :

  • The authentication itself, in order to identify you as a person registered in your corporate directory server.
  • Providing your identity (name, email, department, etc...) to the client application using the OAuth2 protocol
If you are using Windows and the Cerberus server is configured this way, you can log in without entering any login/password, using the Integrated Authentication feature of your browser (enabled by default on Internet Explorer/Chrome, needs to be enabled explicitely with Firefox).

The authentication process

  1. A newcomer in your company, let's call him Cyprien, gets logged on Windows and automatically he obtains a Kerberos Token from your Authentication Server.
  2. He opens his favorite web browser and enters the URL of a client application using Cerberus. He does not know but his personal Kerberos Token is sent into the http requests.
  3. The Client Application transparently redirects Cyprien to Cerberus to request his authentication by forwarding his Kerberos Token. Cerberus will then contact the Authentication Server to find to whom the Kerberos Token is associated. It will then know that the user is Cyprien.

Providing the user identity to the application

The OAuth 2 authorization framework defines the way how a Client Application and a Service Provider (in our case Cerberus) should communicate together to grant access to user's information. There are four ways of granting access, we use the one called Authorization Code.

Some useful definitions first:

  • The Provider is in charge of authenticating a user, has knowledge of user's information and provides HTTP services giving access to user's details - of course by handling permissions.
  • The Client Application does not want to implement the authentication (login/password) on its side, that's why it will ask to the Provider to do it on its behalf. It will then access to the HTTP services to be able to retrieve user's information.

Let's come back to our friend Cyprien.

  1. Once Cyprien has been authenticated by Cerberus, Cerberus generates an Authorization Code. This code is sent back to the Client Application through a callback page that the client application has provided. This code signals to the client application that the user has been successfully authenticated. Usually the client application will require more than that, it needs to know the user name, his e-mail, etc.
  2. In order to retrieve additional information, the client application needs an access token, which enables it to use Cerberus services such as getting user information. To get this token, the client application sends back the Authorization Code to Cerberus. Cerberus checks the code validity, then delivers an access token to the client application.
  3. The client application can now authenticate to Cerberus and retrieve user information, such as its user name, e-mail, department, etc.

Conclusion

Cerberus provides you with an seamless single-sign-on experience, easy to set up and to integrate with your applications.
